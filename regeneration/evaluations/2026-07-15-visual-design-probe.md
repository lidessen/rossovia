# Visual Design Skill Probe

## Claim

[`visual-design`](../../skills/visual-design/SKILL.md) should let
an agent cultivate a source-bearing project direction and then shape, design, or
review a visual relation from the actual object, audience, action, inherited
identity, and material conditions without carrying the Skills site's accepted
treatment into unrelated contexts. It should leave purely functional work to
ordinary development.

The method and interaction runs through the external-user probe used the first
working package name, `aesthetic-cultivation`. Human review then rejected that
name as a public action handle; the later name-migration probe below verifies
the current `visual-design` package. Historical observations retain the working
name where it identifies what actually ran.

The evaluation used fresh read-only Work Cells with the raw tasks and only the
target skill directory in scope. There was no matched no-skill baseline, so the
results support compatible behavior and a useful revision loop but do not
attribute improvement causally to the skill.

## First contrasting-context probe — revision required

**Task:** shape a dense appointment-rescheduling workspace for time-pressured
municipal public-health clerks. The inherited system was described as cool
blue-green, high-contrast, plain-sans, and keyboard-heavy. The task explicitly
asked whether the Skills site's warm-paper, red-black, serif editorial
treatment should transfer.

**Disconfirming observation:** the agent transfers the Skills treatment,
supplies it as a default option, or fails to distinguish shared invariants from
case-specific treatment.

**Observed action:** the agent rejected the Skills treatment and selected
continuity with the stated municipal system. It separated shared identity from
layout, component, and state choices. However, it inferred “no red” as a palette
rule while also leaving red available for error states. A broad identity
description had been promoted into an invented token prohibition.

**Verdict:** core non-transfer boundary supported, but the action probe failed
the source-bound specificity requirement.

**Revision:** the skill and `shape` command now state that a verbal identity
summary is a clue rather than a token specification. Exact colors, typefaces,
component rules, accessibility conformance, and state treatments remain unknown
unless an owning source or artifact establishes them.

The run used 24,146 tokens against a 16,000-token estimate (+50.9%). The
estimate had no declared tolerance, so this is an audit observation rather than
a runtime failure.

## Repeated contrasting-context probe

The task was repeated after the revision and explicitly withheld token,
component, screenshot, and error-state sources.

**Observed action:** the agent again rejected the Skills treatment, retained
only the stated blue-green/high-contrast/plain-sans identity and keyboard-heavy
operating posture, and created distinct sections for open treatment dimensions
and unknowns requiring source or artifact inspection. It declined to prescribe
exact palette values, typeface, spacing, component shapes, error treatment, or
shortcut scheme.

**Verdict:** supported. The skill generated a materially different direction
for a materially different context and did not package the current site's
surface as portable taste.

The repeated run used 33,038 tokens against a 24,000-token estimate (+37.7%).
The additional acceptance condition and explicit unknown-state check increased
both input and output. Future comparable probes should begin from the observed
24k–33k range rather than reusing the original estimate; no hard runtime limit
is warranted from this variance.

## Boundary probe

**Task:** decide whether the skill should own a TypeScript compiler fix that
only corrects a JSON parser return type and has no rendered output,
interaction, document expression, or visual effect.

**Disconfirming observation:** the agent creates an Aesthetic Case, prescribes a
visual treatment, or keeps the task inside the skill.

**Observed action:** the agent quoted the target skill's non-scope, routed the
work to ordinary functional TypeScript development, and explicitly declined an
Aesthetic Case.

**Verdict:** supported.

The boundary run used 24,179 tokens against an 8,000-token estimate (+202.2%).
Most use was input context and repeated skill/Sequence reading. This is poor
economics for a simple routing check and evidence that future boundary tests
should use a smaller evaluator surface; it is not a reason to weaken the
runtime's completion contract.

## Context and authority probe

The action cells loaded `SKILL.md`, `commands/shape.md`, and
`references/concepts.md`; they did not load `design` or `review`. The boundary
cell needed only the main skill. At the time of those probes, the portable
package contained the full one-line Sequence and P16, P07, P05, and P03
interpretations. Both action results left rendering, implementation, and
aesthetic acceptance outside the read-only cell's authority. Later attention
and context-path evidence replaced P03 with P09 in the standing expression team;
the packaged snapshot was regenerated rather than hand-edited.

**Verdict:** supported for progressive disclosure and acceptance separation.

## Project-formation probe

After human review identified the missing movement from admired concrete work
to one prior project direction, the skill added an infrequent `cultivate`
entry. Its lineage replaced P06 with P07: surface/decision separation remains
part of the method, while the selected principle now explicitly owns the move
from concrete sources, through abstraction, back to a new concrete whole.

**Task:** decide what to do for a continuing public-interest neighborhood
health service that will need a website, appointment flow, printed notices, and
illustration, but has no visual direction or human preference decision. The
given admired work was GOV.UK service clarity, transit wayfinding, a historic
community newspaper, and a fashionable luxury wellness app.

**Disconfirming observation:** the agent copies one admired surface, averages
all sources into generic mood words, invents a token kit, or claims to have
accepted a project direction without human preference evidence.

**Observed action:** the agent correctly entered `cultivate`, assigned distinct
source roles, treated the attractive luxury product as a counterexample,
formed three contrasting generative directions with named risks, and stopped
at the exact human choice/recombination needed before synthesizing one
provisional direction. It emitted no palette, typeface, components, or false
acceptance claim.

**Verdict:** supported. This is compatible behavior for the new formation
entry; without a matched pre-revision run, P07's causal contribution remains
unproven.

The run used 47,900 tokens against a 24,000-token estimate (+99.6%), outside
the declared ±60% review tolerance. Inputs accounted for 44,074 tokens. Future
formation probes should estimate from this observation and reduce the evaluator
surface where possible; the variance does not justify a task-time hard limit.

## Existing-system boundary probe

**Task:** an established product already has an accepted design system and
direction; one button's padding differs from its declared component token.

**Observed action:** the agent declined `cultivate`, a new source field, and an
Aesthetic Case. It selected the smallest action: read the owning token and
correct the local padding.

**Verdict:** supported. The project-formation entry did not become a mandatory
preflight.

The run used 36,642 tokens against an 8,000-token estimate (+358%), outside the
declared ±100% review tolerance. The result reinforces the earlier finding that
simple negative routing probes need a materially smaller evaluator context.

## Presentation and progressive-disclosure probes

Human review added two requirements: ship a small official-source reference
base such as Apple and Linear, and make color tokens and layout consequences of
content presentation and attention allocation rather than parallel decoration
choices. It also required common commands and real progressive disclosure.

The stable presentation dependency became:

```text
content and actions
  -> attention path
  -> hierarchy and layout
  -> repeated semantic roles and tokens
  -> case-specific treatment
```

The common operations are now `design` and `review`; low-frequency `cultivate`
and direction-only `shape` remain explicit but are not required stages.

### First design probe — output pass, context revision required

**Task:** design a dense issue-triage screen for a small open-source maintainer,
including content hierarchy, responsive layout, and a color-token approach.

The agent chose `design`, derived attention and layout before semantic tokens,
and did not ask the caller to choose an internal stage. However, it read
`concepts.md` without a concrete ambiguity and did not use the built-in source
index despite absent local design guidance.

**Verdict:** output compatible; progressive disclosure not supported.

The run used 46,130 tokens against a 50,000-token estimate (-7.7%).

### Second design probe — reference use pass, context revision required

After making the fallback condition explicit, the agent selected Linear,
Primer, and GOV.UK and stated transfer boundaries. It still read `concepts.md`
and both the host interpretations and packaged snapshot. This duplicated
authority and context. The source catalog also lived in one file, so selecting
three sources still required loading all source details.

**Revision:** reduce `SKILL.md` from 216 to 169 lines; remove the ordinary-design
pointer to concepts; forbid skill-directory enumeration when routing is known;
make the host/snapshot exclusion explicit; and split the source layer into a
short index plus Apple, Linear, Carbon, Primer, and GOV.UK detail files.

The run used 60,190 tokens against a 46,000-token estimate (+30.8%), within the
declared ±50% tolerance.

### Final context-path probe

The repeated task read exactly:

- host Sequence and selected host interpretations;
- `SKILL.md`, `commands/design.md`, and `presentation-model.md`;
- the short source index; and
- only Linear, Primer, and Carbon detail files.

It did not read the packaged snapshot, concepts, sibling commands, Apple, or
GOV.UK. The output began with content and attention, derived one responsive
layout, introduced semantic roles before values, and named what must not be
copied from the references.

**Verdict:** progressive delivery and reference selectivity supported.

The run used 51,289 tokens against a 60,000-token estimate (-14.5%), within the
declared ±50% tolerance. Its output still promoted several one-consumer spacing
values into a token table, so token admission required a separate revision.

### Token-admission boundary probe

The revised rule admits a token only with two independent consumers or a
declared theme/context substitution. Asked whether one unique card's 13px
illustration-alignment padding should become a token, the agent kept it local
and named the admission rule. It read no source index, concepts, sibling
commands, or packaged snapshot.

**Verdict:** supported. The progressive path can be smaller than the full
presentation model when the command already resolves a bounded token question.

The run used 33,889 tokens against a 30,000-token estimate (+13%), within the
declared ±75% tolerance.

## Site projection probe

The new skill was projected through the same generated `skill-guide` form as
`context-engineering`. The presentation is now declared explicitly rather than
inferred from Starlight's generic Hero state. Astro type checking and static
build passed with zero diagnostics, seven pages built, and all 16 local links
and fragments resolved.

Browser checks at 1280px and 390px showed meaningful content, light theme,
`skill-guide` presentation, no framework error overlay, and no horizontal
overflow. A normal documentation page had no presentation marker, Hero, or
method label. This supports projection containment; it does not make the site's
warm-paper/red-black treatment part of the portable skill.

The Vercel preview was built from uncommitted workspace content while its
version-pinned GitHub URLs necessarily identify the current repository HEAD.
The new skill therefore has no reachable GitHub source at that ref yet. The
preview proves rendering and containment, not final source reachability; commit
and push are required before production promotion can satisfy that boundary.

## External installation and daily-use probe

The earlier documentation described repository-internal slash commands before
the ordinary user path and still used `npx skills install`. The current
[`skills` CLI reference](https://www.skills.sh/docs/cli) defines
`npx skills add <source>`, project scope by default, `--skill` for one selected
skill, and `-g` for a global installation. The portable
[Agent Skills activation model](https://agentskills.io/client-implementation/adding-skills-support#step-4-activate-skills)
also separates model-driven activation from client-specific explicit forms such
as slash commands or mentions. The repository had therefore projected one
client convention as though it were the portable daily interface.

**Installation probe:** `npx skills add . --list` discovered all 12 active
skills and presented the revised user-intent-first description. A fresh
project-level install selected `aesthetic-cultivation`, created the project
skill copy and lock file, and preserved all 20 packaged files, including the
nested design-source and Sequence-snapshot references.

This was historical evidence from before the public-name migration. Directly
testing a local checkout was later retired after the
[self-install alias incident](../../chronicle/records/2026/07/obs-20260715-self-install-source-alias.md)
showed that a repository-local agent-skill symlink could make source and target
the same filesystem state. The current `visual-design` package instead passed
`python3 scripts/probe-skill-installation.py visual-design`: the installer saw
only a disposable source snapshot, wrote to a separate disposable project, and
the probe hash-matched all 20 files before cleanup.

**Ordinary-request probe:** in the fresh project, a Codex request said only
that a documentation site's home, catalog, and content pages did not feel like
one product; it asked for review without edits and explicitly declined an
internal-stage choice. Codex automatically loaded `aesthetic-cultivation`,
selected only `commands/review.md`, retained the read-only boundary, and did not
invent visual findings when the fixture contained no site, screenshot, or URL.
It asked for one of those concrete evidence surfaces and stated the review it
could then perform. No project file was modified.

The probe is behavior evidence for activation, routing, authority, and the
missing-artifact boundary. It is not a token baseline: a globally installed
legacy skill also activated on that machine, and storing the JSONL trace inside
the fixture let the agent read part of its own growing trace. Future cost probes
must isolate the agent home and keep observer logs outside the inspected
workspace.

**Interaction revision:** the skill now treats operation names as internal
routing controls, distinguishes change authority from review authority,
inherits an existing direction without setup, reserves `cultivate` for a
genuine project-formation need, and reports missing host capabilities. The
README now leads with `skills add`, natural-language daily requests, and an
explicit Codex example while labeling explicit activation syntax as
client-owned. The generated public guide exposes the install command and daily
interaction contract as its primary action.

Static build and fragment checks passed. A browser check confirmed meaningful
content, no framework error overlay, the visible “安装与使用” primary action,
and the copyable `npx skills add lidessen/skills --skill
aesthetic-cultivation` command. At 390px the action remained visible and the
page had no horizontal overflow.

## Public-name correction and migration probe

**Observed name failure:** after seeing the installed interface, the designated
human reviewer asked what `aesthetic-cultivation` meant and then rejected it.
The explanation had to translate “cultivation” into ordinary design, redesign,
direction, and review actions. This disconfirmed the name as a sufficient public
handle even though the underlying method remained accepted.

**Selection:** `visual-design` names the common external action. `art-direction`
would understate implementation; `visual-direction` would understate ordinary
design and review. The low-frequency `cultivate` operation remains part of the
method but no longer occupies the install name. The accepted rationale and
operative boundary are retained in
[decision 029](../../design/decisions/029-visual-design-skill.md#name-and-public-entry).

**Migration probe:** after renaming the directory, frontmatter, Codex interface,
source links, site manifest, navigation, decision, and snapshot, `npx skills
add . --list` exposed `visual-design` and no old handle. A fresh project install
copied all 20 files under `.agents/skills/visual-design`; no path containing the
working name remained. The generated site built only
`/docs/skills/visual-design/`, and all 16 local links and fragments passed.
The later disposable, hash-checked probe independently re-established the
current 20-file packaging result without exposing the live source tree to the
installer.

**Ordinary-request probe:** with the freshly installed package and a built site
fixture, a request asked for a visual review without naming a skill or internal
operation. Codex loaded `visual-design`, selected `commands/review.md`, retained
the no-edit boundary, and grounded one recommendation in the built guide. When
the read-only fixture prevented browser runtime setup, it explicitly lowered
the evidence level instead of claiming rendered verification.

**Verdict:** supported for name migration, model-driven activation, review
routing, and authority preservation. It does not prove that the new name alone
caused better behavior. Do not retain an alias or redirect unless an actually
released consumer later creates a migration requirement.

## Post-review guide regression

A later browser review disconfirmed the original guide layout even though the
Astro build and link checks passed. Applying a two-column grid directly to a
Markdown `<li>` made its anonymous text and inline children separate grid
items. On the Improve Agent Workflow guide this stretched the seven-step method
list to 12,938px and each item to 1,640–2,364px, producing a 16,437px desktop
page with large empty columns.

The smallest correction keeps the semantic list intact and positions only the
generated step counter, instead of using the raw list item as a grid container.
In the repeated browser probe at 1280px, the page measured 5,385px, the method
list 1,163px, and its items 158–187px. At 390px the body width equaled the
viewport, the list items measured 276–335px, and no horizontal overflow was
present. These measurements demonstrate closure of the named rendering defect;
they are not universal aesthetic thresholds.

The same review corrected two projection boundaries: the guide now treats
domain vocabulary and runtime entry sections as optional semantic slots, shows
actual `Boundaries`, `Operating boundaries`, or ownership-boundary sections
when present, and derives sidebar guide navigation from the content manifest.
The generated page therefore exposes `Context Engineering`, `Improve Agent
Workflow`, and `Visual Design` together without making the projection a schema
authority over the source skills.

## Deployment decision

Keep `visual-design` active as a two-timescale, progressively disclosed
adaptive method:
infrequent project-level `cultivate`, then case-level `shape`, `design`, and
`review`, with `design` and `review` presented as the common operations. Treat
the first site's treatment as case-local. The next meaningful
behavior evidence is either a human-reviewed cultivation case for a real
project or an authorized `design`/`review` task in a second real product with an
existing identity; do not add fixed tokens, components, style detectors,
browser runtime, or automatic aesthetic acceptance from the present result.
