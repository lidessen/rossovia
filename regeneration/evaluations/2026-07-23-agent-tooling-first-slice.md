# Agent Tooling first-slice probe

**Date:** 2026-07-23

**Status:** structure, local mechanism, action, and boundary claims supported;
Flash-class behavior attribution and real configuration mutation remain
unproven

## Question

Can an independently installed `agent-tooling` skill help an Agent select and
tune a real coding-agent execution surface—including headless CLI use and
tooling reduction—without collapsing into portable environment setup,
project-workflow diagnosis, orchestration, or deletion by installed count?

## Mechanical probes

The zero-dependency cache script was exercised against the locally installed
Codex, Claude Code, and Cursor Agent command families without invoking a model.

- Codex and Claude returned verified versions and help evidence.
- Cursor's executable was found, but its version/help operation failed because
  the local keychain was locked. The script retained the raw failure and
  reported `probe-failed` rather than claiming the tool was absent or usable.
- A valid, source-linked Codex capability projection passed `put`, appeared
  through `show`, and kept documentation freshness at
  `recorded-not-revalidated`.
- An invalid projection with a malformed timestamp and no sources failed
  closed.
- The Sequence snapshot check, disposable installation probe, site content
  projection, and Astro check passed.

These observations support the cache as a rebuildable evidence projection.
They do not establish any vendor capability beyond the captured version/help
and separately checked sources.

## Action probe

A fresh low-cost independent Agent received the packaged skill and this raw
scenario: automate a read-only repository summary through Claude Code while an
environment discovers forty skills, five MCP servers, and seven hooks; the
maintainer proposes deleting everything unused.

The Agent:

- selected a fresh read-only headless run rather than an interactive TUI;
- kept the ordinary environment as baseline and used the currently documented
  clean/bare form only as a diagnostic counterfactual;
- rejected deletion from installed count;
- held repository revision, task, model/auth carrier, permissions, and
  acceptance constant;
- compared correctness, citations, time/usage, hook errors, tool activity,
  missing context, and recovery effort; and
- required current installed help, official documentation, effective loaded
  surfaces, permissions, and output behavior before constructing exact
  commands.

This supports the action claim: the skill enabled a concrete headless and
tuning decision without turning volatile flags into doctrine.

## Boundary probe

A second fresh independent Agent received two requests:

1. set up Codex, Claude Code, Cursor, preferences, and skills on a second
   laptop; and
2. split a large refactor among eight subagents and aggregate results.

It routed the first to `agent-environment`, which owns non-secret portable
desired state and cross-device reconciliation. It routed the second to Work
Cell or another orchestrator because delegation and aggregation are
composition, not an execution-carrier choice.

This supports the two highest-risk ownership boundaries.

## Flash probe limitation

A read-only Work Cell probe was prepared for the OpenCode Go
`deepseek-v4-flash` route. The sandboxed attempt contacted no provider and
settled with zero tokens because network access was unavailable. The requested
network rerun was rejected by the approval boundary because it would send
repository-derived skill and principle content to an external provider without
specific payload authorization. No workaround was attempted.

The independent action and boundary probes therefore used a low-cost local
validation Agent, not the conservative Flash-class reference model. The first
slice must not claim Flash stability until an explicitly authorized,
payload-bounded comparison runs.

## Decision

Retain `agent-tooling` as the first slice:

- its name and scope cover both coding-agent operation and evidence-based
  tuning;
- it remains distinct from environment migration, workflow diagnosis,
  orchestration, and model evaluation;
- the local script carries deterministic probe/cache mechanics without making
  the skill depend on Rossovia Workbench; and
- live vendor evidence remains outside stable doctrine.

Before using the skill to remove real user tooling, run an authorized matched
normal/lean task through that harness and preserve a rollback. Before claiming
Flash-class reliability, run the held-out action and boundary cases through an
approved Flash route without leaking unrelated repository context.
